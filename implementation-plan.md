# Implementation Plan — Growee Family Launch

> Roadmap по чанкам. Каждый чанк = отдельный диалог Claude Code / Codex. Старт нового диалога: `@implementation-plan.md запусти Chunk N`. После завершения чанка — commit + обновить «Прогресс» ниже.

## Цель кампании

Довести Growee до состояния, в котором им можно реально пользоваться в семье владельца. Аудит и фикс существующего, **не** добавление новых фич. Web push, invite flow, переименование продукта, геймификация — всё после того, как первая семья начнёт пользоваться.

Definition of done всей кампании: владелец и его семья пользуются приложением на ежедневной основе минимум неделю, без блокирующих багов.

---

## Chunk 1 — Smoke audit

**Что делаем:** запустить `npm run dev`, пройтись по всем флоу в обеих ролях (parent + child), задокументировать всё видимо сломанное в чеклист. **Никаких правок кода в этом чанке.**

**Маршрут прохождения:**

1. `/auth` — регистрация нового пользователя, логин существующего
2. Family setup — создание семьи, добавление ребёнка
3. Parent: `/parent` (Dashboard) → `/parent/children` → `/parent/tasks` → `/parent/templates` → `/parent/store` → `/parent/jobs` → `/parent/profile` → `/parent/security`
4. Child: переключение на роль / выбор `currentChild` → `/` (Today) → `/schedule` → `/exchange` → `/store` → `/rewards` → `/achievements`
5. Сквозной сценарий: родитель создаёт task template → ребёнок видит на Today → ставит шаги → балансы обновляются → родитель видит транзакцию

**Результат чанка:** файл `audit-findings.md` в корне (создаётся в этом чанке) с таблицей: `[флоу] [симптом] [серьёзность: blocker/major/minor] [подозреваемая причина]`.

**Definition of done:** все 5 областей пройдены, чеклист записан, ни строчки кода не изменено.

---

## Chunk 2 — Schema & RLS audit

**Что делаем:** проверить, что БД действительно делает то, что ожидает фронтенд.

**Что проверить:**

1. **RLS-политики** — для каждой таблицы убедиться, что политики соответствуют сценариям (родитель видит только свою семью, ребёнок видит только свои данные, write-операции корректно ограничены)
2. **Триггеры** — баланс (`children.balance`) обновляется триггером после insert в `transactions`? Транзакция автоматически создаётся при `task_instances.state → 'done'`? Проверить наличие и корректность
3. **Foreign keys и каскады** — что происходит при удалении ребёнка / семьи / task_template
4. **Висячие сущности** — таблицы, на которые никто не ссылается из хуков, или хуки, ссылающиеся на отсутствующие колонки/функции
5. **Сравнение схемы и хуков** — каждый `useX.ts` хук → проверить, что его SELECT/INSERT/UPDATE поля существуют в `00000000000000_initial.sql`

**Результат чанка:** дополнить `audit-findings.md` секцией «Schema/RLS issues», список расхождений между ожиданием фронта и реальностью БД.

**Definition of done:** проверены все ~20 таблиц, расхождения записаны.

## Chunk 3 — Per-flow fixes

**Что делаем:** взять `audit-findings.md`, идти сверху вниз по серьёзности (blocker → major → minor), фиксить минимальными правками. Каждый блокирующий баг — отдельный коммит.

**Группы (по областям):**

- **A.** Auth / Family setup
- **B.** Tasks (templates, instances, steps, generation)
- **C.** Store / Wishlist / Purchases
- **D.** Job board / Job claims
- **E.** Schedule / Day templates / Achievements

**Если** во время фиксов вылезает архитектурное решение — остановиться, обсудить с владельцем, не догадываться.

**Definition of done:** все blocker и major из чеклиста закрыты. Minor — по усмотрению (можно перенести в backlog). После каждой группы — `npm run build && npm run lint` зелёные.

## Chunk 4 — Pre-launch hardening

**Что делаем:** базовая гигиена перед тем, как пускать семью.

**Список:**

1. Создать `.env.example` с `VITE_SUPABASE_URL` и `VITE_SUPABASE_PUBLISHABLE_KEY` (без значений, только имена)
2. Удалить `bun.lockb` (используется npm, лишний lockfile путает)
3. Добавить минимальный smoke E2E на happy path: логин → создать задачу → ребёнок выполнил → баланс обновлён. Стек: Playwright (один файл, без CI). Если за 2 часа не получается — отложить, не блокировать запуск
4. Проверить production build на Vercel — открыть deploy preview, прокликать Today/Schedule/Store
5. Мини-README: как запустить локально, как добавить переменные

**Definition of done:** `.env.example` есть, `bun.lockb` удалён, prod-сборка на Vercel рабочая.

## После запуска (post-family-launch backlog)

Сюда складываем то, что **не** в скоупе текущей кампании:

- **Web push + invite flow** — для своей семьи аккаунты заводятся вручную, инвайты не нужны. Понадобится при коммерческом запуске.
- **Node 20.19+ upgrade и Vite 7/8** — текущая машина на Node 20.11, vite зафиксирован на 6.x. Когда обновится Node — пересмотреть вверх. Уязвимостей в текущей конфигурации нет, это чисто гигиена.
- **Консолидация имени `Growee`** — переименовать Supabase project (`kids-routin-app` → `growee`), Vercel project (`kids-routine-app` → `growee`), GitHub repo (`routine-star-kids` → `growee`). Каждое — ручная операция в UI соответствующего сервиса, требует обновления `.env`/CI/links. Не делаем под запуск семьи.
- **Геймификация / «игра в жизнь»** — прокачка навыков (ответственность, внимание, целеполагание, финансы, налоги, накопления). Большая отдельная ветка, спланируется после first-use feedback.
- **TypeScript strict mode** — включить постепенно, файл за файлом.
- **CI/тесты** — после первого smoke E2E развить в полноценный suite, добавить GitHub Actions.
- **`.env.example`-разнообразие** — если появятся новые секреты (web push VAPID-ключи, payment provider) — обновлять там.
